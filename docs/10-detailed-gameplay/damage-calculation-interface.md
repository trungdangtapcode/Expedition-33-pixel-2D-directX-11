# Damage Calculation System & Architecture

## Overview
The Damage Calculation system decouples the raw damage logic (Physical vs Magical calculations, Multipliers, Flat Bonuses) from the base `IBattler` interface. Originally, damage formulas (like `Effective = Raw - DEF`) were hardcoded inside `Combatant::TakeDamage`. This was replaced by an extensible, interface-driven `IDamageCalculator` implementation.

## Core Components

1. **`DamageRequest` Struct**
   A data payload encapsulating all necessary context for a hit:
   - `attacker` / `defender` (pointers to `IBattler`).
   - `DamageType` enum (`Physical`, `Magical`, `TrueDamage`).
   - `skillMultiplier` (float scale, e.g., 2.0x for Rage Burst).
   - `flatBonus` (flat integer addition before defenses).

2. **`DamageResult` Struct**
   The finalized output generated by the calculator:
   - `effectiveDamage` - The actual amount subtracted from HP.
   - `rawDamage` - The pre-defense damage (used for logging or special UI).
   - `defenseUsed` - The amount mitigated (used for logs/UI).
   - `isCritical` - Reserved for future RNG critical strikes.

3. **`IDamageCalculator` & `DefaultDamageCalculator`**
   - **`IDamageCalculator`**: A pure virtual interface defining the mathematical contract (`virtual DamageResult Calculate(const DamageRequest& request) const = 0`).
   - **`DefaultDamageCalculator`**: The concrete JRPG implementation. Calculates logic dynamically by cross-referencing `DamageType` (e.g., matching Physical to `atk/def`, Magical to `matk/mdef`).

## The Turn-Based Damage Pipeline
1. **Skill Execution**: When an ability triggers (e.g., `AttackSkill`), it constructs a `DamageRequest` with context (e.g. `Physical`, `1.0f` multiplier).
2. **Action Delay**: The `AnimDamageAction` encapsulates this request. It waits until the visual 2D sprite frame hits its designated `damageMoment` (e.g. 80% through the sword swing).
3. **Calculation Moment**: At the exact frame of the impact, `DefaultDamageCalculator::Calculate()` evaluates the live stats (factoring in any debuffs/buffs that might exist at that exact split-second).
4. **Application**: `TakeDamage(DamageResult)` is finally triggered, modifying HP and feeding back standard Rage rewards (without recalculating armor).

# Post-Mortem Report: The "Verso Turn View Flood" Bug
*Or: How updating the Damage System broke the UI animation queue entirely.*

### The Incident
Immediately after refactoring the codebase to support Magical Damage attributes (`matk`, `mdef`), the Battle UI entirely broke. Upon launching the game, the top-left Turn View UI was populated entirely by the protagonist (Verso)'s portraitâ€”effectively showing Verso taking 20+ turns in a row without enemies responding.

### The Root Cause: C++ Brace Initialization Trap
In the C++ language, **Aggregate Brace Initialization** maps values strictly based on the order members are declared in the struct memory layout.

**Old Struct (`BattlerStats.h`):**
```cpp
int hp, maxHp;
int mp, maxMp;
int atk, def, spd;   // Index 6 is SPD
int rage, maxRage;   // Index 8 is MaxRage
```

**Old Party Manager Call:**
```cpp
BattlerStats mVersoStats { 100, 100, 50, 50, 25, 10, 10, 0, 100 }; 
// Value '10' maps to SPD. Value '100' maps to MaxRage.
```

When building the damage system, new variables were naturally inserted into the stats struct to keep offensive and defensive stats together:

**New Struct (`BattlerStats.h`):**
```cpp
int hp, maxHp;
int mp, maxMp;
int atk, def; 
int matk, mdef;      // !! NEW VARIABLES INSERTED HERE !!
int spd;             // Index 8 is now SPD!
int rage, maxRage;
```

Because `mVersoStats` was hard-initialized in `PartyManager.h` using comma-separated values, the shift in structure completely desynced the memory layout mapping. The system mapped the parameters sequentially: `matk = 10`, `mdef = 0`, and the 100 originally intended for `maxRage` fell squarely into `spd`.

**The resulting stats for Verso:**
- **Skeletons' Base SPD:** 5
- **Verso's accidental SPD:** 100

### Why does this cause a Turn View bug?
The Battle Engine runs on an **Action Value (AV)** tick system (using the formula: `10000.0f / SPD = ticks_until_next_turn`).
- **Skeleton AV requirement:** 2000 points.
- **Verso's AV requirement:** 100 points.

Because Verso was moving 20x faster than the Skeletons, the `BattleManager::GetFutureTurnQueue` correctly simulated the timeline and filled the upcoming `mNodes` purely with Verso's portrait because mathematically, he really *was* scheduled to go twenty times before the first monster.

### The Fix & Takeaway
The `PartyManager.h` array was expanded mapped line-by-line with descriptive comments indicating exactly which field maps to which integer:
```cpp
BattlerStats mVersoStats {
    100, // hp
    100, // maxHp
...
    25,  // matk
    10,  // mdef
    10,  // spd
...
};
```
**Architecture Rule enforced**: Always use explicit mapping, explicit constructors, or data-driven JSON injection instead of blind brace assignment if a C++ struct is subject to continuous iteration.
